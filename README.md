# pxpipe-demo

This repo contains a small, reproducible pxpipe experiment inspired by the video ["NEW Fable 5 API Usage Exploit (Reduce Costs Immediately)"](https://www.youtube.com/watch?v=dzfFN0RgPlI).

The video's core idea is that some large static text contexts may be cheaper to send as rendered images than as normal text tokens. This repo tests that idea in a simple 2x2 benchmark: direct text input versus pxpipe image-compressed input, on an easy copying task and a harder ledger aggregation task.

## What Was Tested

The experiment asks one practical question:

> Can pxpipe reduce input tokens while preserving answer quality?

The test corpus is a dense JSONL ledger:

- 12 JSONL files
- 498,775 bytes total, about 487 KB
- roughly 500 KB of static text context per run

pxpipe rendered that static context into PNG images for the compressed runs:

- easy pxpipe run: 19 PNGs, 1,887,206 bytes
- hard pxpipe run: 18 PNGs, 1,878,622 bytes
- total included dumps: 37 PNGs, 3,765,828 bytes

## The 2x2 Design

There are two tasks:

1. Easy answer-card task: the expected JSON answer is placed inside the static context and the model only needs to copy it.
2. Hard ledger task: the model must aggregate the JSONL ledger and compute final tenant state, region counts, top services, and high-budget tenants.

Each task is run two ways:

| Run | Task | Route |
|---|---|---|
| `direct_easy` | Easy answer-card task | Direct OpenAI API |
| `pxpipe_easy` | Easy answer-card task | Through pxpipe |
| `direct_hard` | Hard ledger aggregation | Direct OpenAI API |
| `pxpipe_hard` | Hard ledger aggregation | Through pxpipe |

## Result Summary

Model requested: `gpt-5.5`

Model reported by the API: `gpt-5.5-2026-04-23`

| Task | Direct accuracy | pxpipe accuracy | Direct input tokens | pxpipe input tokens | Input token savings |
|---|---:|---:|---:|---:|---:|
| Easy answer-card task | Pass | Fail | 152,655 | 24,371 | 84.04% |
| Hard ledger task | Fail | Fail | 152,153 | 24,362 | 83.99% |

The important interpretation is that pxpipe saved a lot of input tokens, but accuracy did not survive this specific setup. The easy task is the cleanest signal: direct text passed, pxpipe failed. The hard task is less diagnostic because direct text also failed.

## Cost Estimate

Using standard `gpt-5.5` pricing from OpenAI's API pricing page, and treating the direct runs as long-context calls and pxpipe runs as short-context calls, the final four-call 2x2 run cost about:

- direct calls only: about $3.16
- pxpipe calls only: about $0.28
- all four calls: about $3.44

That means the pxpipe route was about 91% cheaper for the two comparable calls in this run. But the cheaper route was not useful by itself because the pxpipe answers were wrong.

Pricing reference: https://developers.openai.com/api/docs/pricing

## Files

- `pxpipe_showcase/EXPERIMENT_REPORT.md`: detailed plain-English report.
- `pxpipe_showcase/comparison_summary.json`: compact machine-readable run summary.
- `pxpipe_showcase/golden_answer.json`: expected correct answer.
- `pxpipe_showcase/answers/`: four model answer files from the final 2x2 run.
- `pxpipe_showcase/evidence/events.jsonl`: pxpipe event log for the two compressed calls.
- `pxpipe_showcase/dumps/`: PNG images generated by pxpipe for the two compressed calls.
- `pxpipe_showcase/corpus/`: generated dense JSONL corpus.
- `pxpipe_showcase/generate_showcase.mjs`: corpus and golden-answer generator.
- `pxpipe_showcase/test_prompt.md`: hard aggregation prompt.

Excluded on purpose:

- `.env` files
- API keys or secrets
- local machine paths
- earlier exploratory model responses
- 4xx request bodies from failed setup attempts
- run logs

## Reproduce the Corpus

From the repo root:

```bash
node pxpipe_showcase/generate_showcase.mjs
```

This regenerates the JSONL corpus, `golden_answer.json`, `test_prompt.md`, and the local showcase README.
